PAN Format
What a Valid PAN Number Looks Like
A PAN number is exactly 10 characters — 5 letters, 4 digits, 1 letter. Example: AHGVE1276F
A H G V E5 Uppercase Letters
1 2 7 64 Digits
F1 Letter

First 5 Characters — Uppercase Letters

Must be alphabetic A–Z. No adjacent characters can be the same. Cannot form a sequential pattern.

✓ AHGVE — Valid  ✗ AABCD — Adjacent repeat  ✗ ABCDE — Sequential

Next 4 Characters — Digits Only

Must be numeric 0–9. No adjacent digits can be the same. Cannot form a sequential pattern.

✓ 1276 — Valid  ✗ 1123 — Adjacent repeat  ✗ 1234 — Sequential

Last Character — Single Uppercase Letter

The 10th and final character must be an alphabetic uppercase letter A–Z.

✓ F — Valid  ✗ 9 — Digit, not letter

Exactly 10 Characters

Any PAN shorter or longer than 10 characters is automatically invalid regardless of content.

✓ AHGVE1276F — 10 chars  ✗ AHGVE127 — Only 9 chars
The Full Story
Problem → Solution → Real Business Impact
What the data problem was, how SQL solved it, and what the output delivers.

The Data Quality Problem

A dataset of 10,000 PAN numbers arrived with multiple quality issues making it impossible to use directly for compliance or reporting purposes without a structured cleaning pipeline.

  • 965 Missing / Null Records

    Nearly 10% of the dataset had null or empty PAN values — these needed to be identified, counted, and excluded from validation without losing the audit trail.

  • Duplicate Entries

    9 duplicate PAN numbers were present — counting them multiple times would inflate valid counts and produce incorrect compliance reports.

  • Inconsistent Formatting — Spaces & Lowercase

    Many records had leading/trailing spaces (e.g. "hyuij7902r ") and lowercase letters (e.g. "hvofe5635y") that would fail validation even if structurally correct.

  • No Standard Validation Logic Existed

    The standard PAN regex alone isn't sufficient — you also need to check for adjacent repeating characters and sequential patterns, which required custom SQL functions.

How SQL Solved It — Step by Step

Built a complete SQL pipeline: stage table → cleaning → custom validation functions → categorization view → summary report. Every step is auditable and reproducible.

  • Stage Table → Clean Table

    Raw data loaded into a staging table. A cleaned table is created using DISTINCT + UPPER() + TRIM() — removing duplicates, fixing case, and stripping spaces in one query.

  • Custom Function — Adjacent Character Check

    A custom SQL function loops through each character pair and returns TRUE if any two adjacent characters are the same — catching invalid patterns like AABCD or 1123.

  • Custom Function — Sequential Pattern Check

    A second function checks if characters form a sequential pattern (ABCDE, 1234 etc.) by comparing ASCII values of consecutive characters — returning TRUE if sequential.

  • Validation View + Summary Report CTE

    A SQL View categorises every cleaned PAN as Valid or Invalid. A CTE-based summary report then gives total processed, valid, invalid, and missing counts in one query.

One query creates the clean table: DISTINCT removes duplicates, TRIM() strips spaces, UPPER() fixes case — all null and blank rows excluded.

Loops through character pairs using SUBSTRING() — returns TRUE if any adjacent pair matches. Used to catch AABCD (letters) and 1123 (digits).

Custom Function — Sequential Pattern Check

Checks ASCII differences between consecutive characters — if every difference is exactly 1, it's sequential. Catches ABCDE and 1234 type patterns.

6

A SQL View using CTEs applies all rules: regex format check, both custom functions on letter and digit segments. Every PAN gets a "Valid PAN" or "Invalid PAN" label.

7

Summary Report — Final CTE Query

A single CTE query produces the complete audit report: total processed, valid count, invalid count, and missing/incomplete PANs calculated from the stage table.

SQL Code
Key Queries from the Project
The most important SQL written for this pipeline.
Step 1 — Create Cleaned Table 
CREATE TABLE pan_numbers_dataset_cleaned
AS
SELECT DISTINCT
    UPPER(TRIM(pan_number)) AS pan_number
FROM stg_pan_numbers_dataset
WHERE pan_number IS NOT NULL
  AND TRIM(pan_number) <> '';
-- Removes: nulls, blanks, duplicates, fixes case & spaces in one query
Step 2 — Custom Function: Adjacent Repetition Check 
CREATE OR REPLACE FUNCTION fn_check_adjacent_repetition(p_str TEXT)
RETURNS BOOLEAN
LANGUAGE plpgsql
AS 538
BEGIN
    FOR i IN 1 .. (LENGTH(p_str) - 1) LOOP
        IF SUBSTRING(p_str, i, 1) = SUBSTRING(p_str, i+1, 1)
        THEN
            RETURN TRUE;  -- Adjacent repeat found
        END IF;
    END LOOP;
    RETURN FALSE;  -- No adjacent repetition
END;
538
-- Usage: fn_check_adjacent_repetition('AABCD') → TRUE (invalid)
--        fn_check_adjacent_repetition('AHGVE') → FALSE (valid)
Step 3 — Custom Function: Sequential Pattern Check 
CREATE OR REPLACE FUNCTION fn_check_sequence(p_str TEXT)
RETURNS BOOLEAN
LANGUAGE plpgsql
AS 538
BEGIN
    FOR i IN 1 .. (LENGTH(p_str) - 1) LOOP
        IF ASCII(SUBSTRING(p_str, i+1, 1)) - ASCII(SUBSTRING(p_str, i, 1)) <> 1
        THEN
            RETURN FALSE;  -- Not sequential
        END IF;
    END LOOP;
    RETURN TRUE;  -- All characters are sequential
END;
538
-- Usage: fn_check_sequence('ABCDE') → TRUE (invalid — sequential)
--        fn_check_sequence('AHGVE') → FALSE (valid — not sequential)
Step 4 — Validation View: Valid vs Invalid Categorization 
CREATE OR REPLACE VIEW vw_valid_invalid_pans AS
WITH cte_cleaned_pan AS (
    SELECT DISTINCT UPPER(TRIM(pan_number)) AS pan_number
    FROM stg_pan_numbers_dataset
    WHERE pan_number IS NOT NULL AND TRIM(pan_number) <> ''
),
cte_valid_pan AS (
    SELECT * FROM cte_cleaned_pan
    WHERE
        fn_check_adjacent_repetition(pan_number) = 'false'
        AND fn_check_sequence(SUBSTRING(pan_number,1,5)) = 'false'
        AND fn_check_sequence(SUBSTRING(pan_number,6,4)) = 'false'
        AND pan_number ~ '^[A-Z]{5}[0-9]{4}[A-Z]$'
)
SELECT
    cln.pan_number,
    CASE WHEN vld.pan_number IS NULL
        THEN 'Invalid PAN'
        ELSE 'Valid PAN'
    END AS status
FROM cte_cleaned_pan cln
LEFT JOIN cte_valid_pan vld ON vld.pan_number = cln.pan_number;
Step 5 — Final Summary Report 
WITH cte AS (
    SELECT
        (SELECT COUNT(*) FROM stg_pan_numbers_dataset) AS total_processed_records,
        COUNT(*) FILTER (WHERE vw.status = 'Valid PAN')   AS total_valid_pans,
        COUNT(*) FILTER (WHERE vw.status = 'Invalid PAN') AS total_invalid_pans
    FROM vw_valid_invalid_pans vw
)
SELECT
    total_processed_records,
    total_valid_pans,
    total_invalid_pans,
    total_processed_records - (total_valid_pans + total_invalid_pans) AS missing_incomplete_pans
FROM cte;
